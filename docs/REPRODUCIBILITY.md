# Reproducibility (REQ-DOC-004, REQ-GOV-050)

Every deterministic vector and figure the project publishes is regenerated by code, not
hand-written. `reproduce` regenerates each vector and diffs it against the committed
expected set; any mismatch is a non-zero exit and a failed CI gate.

## Running

```
cargo run --release --bin overlay-broadcast -- reproduce
```

This regenerates and checks the deterministic vectors (`cli::reproduce::generate_vectors`):
SHA-256d / SHA-256 / hash160 known values and a merkle root over fixed leaves. A correct
build reproduces them byte-for-byte; a tampered expected value is detected
(`TST-CLI-003`).

## Benchmark figures (REQ-BENCH-001/003)

Benchmark *figures* are this system's own measurements on the stated host, never an
external capacity number restated as a result. The **deterministic** parts of each study
— communique counts, encrypted-data-item counts, subscription transaction counts
(`bench::deterministic_results`) — are reproducible and diffable; only host-specific
latency varies and is labelled as a measurement. `TST-BENCH-001` asserts the deterministic
counts are identical across runs.

## Determinism sources

- CKD derivation is a pure function of (seed, position) — `TST-TST-002`.
- The reproduction PRNG (fuzz corpus, `fuzzprop`) is a fixed-seed SplitMix64, so any crash
  is a committable regression seed.
- Builds pin the toolchain (`rust-toolchain.toml`, Rust 1.96.0) and all dependency
  versions (`Cargo.lock`); reproducibility build flags are documented in
  `docs/ARCHITECTURE.md`.
